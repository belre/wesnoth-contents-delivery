# Wesnothアセット利用ライセンス調査メモ

最終更新: 2026-07-09

## 1. 大前提：Wesnothの画像ライセンスは単一ではない

- Wesnoth公式Wiki「Wesnoth:Copyrights」により、画像・音声アセットは
  **GNU GPL v2 or later** か **CC BY-SA 4.0** のいずれかでライセンスされる。
- 既存の古い素材（目安: 2017年7月30日より前に追加）は GPL v2+ が多い。
- それ以降の新規貢献は CC BY-SA 4.0 がデフォルト。ただし既存GPL素材の
  「クリーンアップ」的改変（パレット調整等）は派生物とみなされ、
  明記がなければ GPL v2+ を引き継ぐと保守的に解釈するのが安全。

## 2. ライセンス判定の一次情報源

- **最も確実**: 画像ファイルのEXIFタグ（`Artist` / `Copyright` / `UserComment`）。
  Wesnothプロジェクトは2024年（Issue #8863）に `copyrights.csv` 管理から
  EXIF埋め込み方式へ移行済み。CIで `Copyright` タグが
  `GNU GPL v2+` または `CC BY-SA 4.0` のいずれかであることを強制。
- **次善策**: `git log --follow` によるコミット履歴確認。
  ただしメタデータのみの一括変更コミット（例: PR #10229, XMP→EXIF移行）を
  「内容変更」と誤認しないよう除外が必要。
- 実測結果: sprites/ 配下を `scan_sprite_licenses.py` で走査したところ、
  **約80%が GPL v2+、残り約20%が CC BY-SA 4.0 / 要確認**という比率だった。
  → 3節の通りGPLv3統合方針を採る限り、この比率自体は実務上の障害にはならない。
  ただし「アセットプール全体の8割がGPL v2+由来」という規模感は、
  PSP/原本リポジトリ側の管理対象がかなり大きくなることを意味するため、
  6節の運用フロー（画像+PSP=GPLv3別リポジトリ）が前提として重要になる。

## 3. 各要素のライセンス互換性

### 3-1. GPL v2+ → GPLv3への統合
- Wesnothの表記は一貫して「GPL v2 *or later*」。
  "or later" 条項によりGPLv3の傘下に統合可能。

### 3-2. CC BY-SA 4.0 → GPLv3への統合
- 2015年10月、Creative CommonsとFSFが正式に
  **CC BY-SA 4.0 から GPLv3への一方向互換性** を宣言済み。
  (https://creativecommons.org/2015/10/08/cc-by-sa-4-0-now-one-way-compatible-with-gplv3/)
- 逆方向（GPLv3 → CC BY-SA）は不可。
- CC BY-SA側の表示義務（TASL: Title/Author/Source/License）は
  GPLv3化しても消えない。改変者はBY-SAの表示要件も満たす必要がある。

### 3-3. 結論
→ **プロジェクト全体をGPLv3で頒布すると宣言すれば、GPL v2+ と CC BY-SA 4.0
  の両方の画像を法的に矛盾なく統合できる。**

## 4. コード側をGPLに巻き込まないための設計

### メカニズム①：派生物にならない（Word型）
- 「データを表示するだけのプログラム」は、そのデータの派生著作物にならない。
- MS WordがGPL小説を開いても、WordがGPL化されないのと同じ理屈。
- ブラウザ/ゲームエンジンが画像を「表示するだけ」なら、ほぼ無条件に安全。

### メカニズム②：アグリゲート条項（民生機器型）
- GPLv3 第5条: 対象作品と「結合して一つの大きなプログラムを形成しない」
  別個独立の作品との編集物は「アグリゲート」であり、
  GPLは他の部分に適用されない。
- 条件: 画像データをコードに埋め込まない（Base64インライン化等は避ける）。
  静的ファイルとして分離し、実行時に読み込む設計を維持すること。
- 過去のGPL訴訟（Cisco/Linksys, Samsung, VIZIO等）の多くは、
  この「アグリゲートのつもりが実は結合していた」ケース。

### 誤解しやすい点
- **「Vercel vs CDN」は本質ではない**。Vercelの静的配信もCDN配信の一種。
  問題はビルド工程で画像がJSバンドルに埋め込まれるかどうかの一点。
- **「ビルド済みだけ公開し元ソースは非公開」は成立しない**。
  GPLv3の義務は「頒布したオブジェクトコードに対応するソース」を
  提供したかどうかで判定される。コード自体がアグリゲートとして
  分離されていない限り、この回避は単純なGPL違反になる。

## 5. GPLv3の頒布義務（画像側）の満たし方

- GPLv3 第6条(d): オブジェクトコードを「指定された場所からコピーできる形」で
  頒布する場合、**同じ場所から対応するソースへの同等アクセスを提供**すれば
  頒布義務を満たしたことになる。
- 実装案:
  - CDN配信する画像＝「オブジェクト」の頒布経路
  - 別リポジトリで公開する原本/PSPファイル＝「ソース」の頒布経路
  - サイトのクレジットページ等から両者を明確にリンクさせる

## 6. 採用する運用方針（最終形）

1. **アプリケーションのコードリポジトリ**: 当面は **No License**（LICENSEファイルを
   付けない）で公開する。理由:
   - No Licenseなら閲覧はできても第三者に利用・改変・再配布の権利は発生しない
     （all rights reservedがデフォルト）。
   - MIT等を一度でも付けて頒布すると、その時点のスナップショットについては
     取得者の権利を不可逆に手放すことになる（後から撤回不可）。
   - 単独著作者である間は「いつでもMIT等に切り替えられる自由」自体に価値がある。
     製品として公開する段階で、その時点の判断としてMIT化するかを改めて決める。
   - 画像データをソースに埋め込まない設計は引き続き徹底する。
2. **CDN配信する画像＋PSPファイル**: GPL v2 or later（将来的にGPLv3で統合）
   として公開。改変可能形式（PSP）へのアクセスをセットで提供する。
3. **加工スクリプト自体**: 独自実装であればGPLの影響を受けない。
   公開は任意（画像データそのものとは別問題）。
4. **配信方式**: Vercel/CDNいずれでも良いが、画像をJSバンドルに
   埋め込まず、静的ファイルとして分離して配信することが必須条件。

## 6-2. 最終構成（確定事項）

### リポジトリ構成: 3つ

| リポジトリ | 公開状態 | ライセンス | 用途 |
|---|---|---|---|
| 本体エンジン（旧 `wesnoth-react-engine`） | **非公開** | 判断不要（非公開のため選択不要） | react-expo-wesnoth等、フル機能の実装本体 |
| `mini-wesnoth`（本体からの切り出し） | 公開 | No License | デモ・ポートフォリオ用の縮小公開版 |
| 画像・PSPリポジトリ | 公開 | GPLv3 | 両者から共用する素材一式 |

- 当初「コード1本化」で検討していたが、「mini-wesnothだけ将来的に
  切り出して公開したくなる可能性」を考慮し、本体エンジンは完全非公開、
  公開用の縮小版として `mini-wesnoth` を別途切り出す方針に変更。
- 本体エンジンは非公開である以上、LICENSE選択自体が不要
  （見えない以上、No License/MITのどちらであるかは無意味）。
- `mini-wesnoth` は公開するため、これまで通りNo License方針を維持。

### 切り出し作業時の注意点
- ディレクトリコピーではなく、**git履歴を持ち越さない新規リポジトリ**
  として書き出すこと（コミットメッセージに本体側の内部設計が
  残っている可能性があるため）。
- コメント・変数名・設定ファイルに、本体エンジンや
  react-expo-wesnoth固有のロジックへの言及が残っていないか確認する。
- 画像・PSPをコードに埋め込まない設計は `mini-wesnoth` 側でも維持し、
  GPLv3アグリゲート条項の要件を引き続き満たすこと。

### 構成原則（変更なし）
- ホスティング先（Vercel / CloudFront）は問わない。
  どちらも静的ファイル配信＝CDN配信として扱えるため整合性は同じ。
- 画像・PSPファイルはコードリポジトリと分離した独立リポジトリで
  GPLv3として保管し、公開・非公開いずれのアプリからも共用する。
- アプリ側コードは画像バイト列を埋め込まず、`public/` 経由の
  静的配信に徹することで、コード側がGPLに巻き込まれない設計を維持する。
- 過去のNo License/MIT混在リポジトリ（画像非同梱・.gitignoreでローカル運用）
  は、Wesnothアセットのリスクを一切含まないことを確認済み。

## 8. 商標リスク（ライセンスとは別の論点）

- 「BATTLE FOR WESNOTH」はWesnoth, Inc.が米国で商標登録済み
  （登録番号5109317、出願2016年5月17日）。対象はゲームソフトウェア全般。
- GPL/CC BY-SAのライセンスは**著作権**の許諾であり、**商標**の使用許諾
  ではない。ライセンスに完全準拠していても、プロダクト名に
  "Wesnoth" を含めることについては別途商標リスクが残る。
- 想定される現実的なリスクは「ライセンスを変えてほしい」という要求
  ではなく、**「プロダクト名からWesnothを外してほしい」という
  商標由来の要求**である可能性が高い。
- 対応策:
  - プロダクト名に"Wesnoth"を直接含めない
    （`mini-wesnoth` 等の名称は将来的に変更を検討）
  - README等で「Battle for Wesnothのアセットを使用」という
    説明的言及は問題になりにくい（nominative fair use）
  - ロゴ・ブランドイメージの模倣は避ける
